# Claim-Safe Artifact Engineering for Software-Only EDA Experiments: Design and Evaluation of ArtifactGate-EDA

## Abstract

Electronic design automation studies often distribute experiment files,
simulation logs, synthesis reports, and reproduction scripts without a
machine-checkable connection between those artifacts and the claims made about
them. This creates a review problem: an artifact package may be complete as a
file bundle while still allowing claims that exceed the available software-only
evidence. ArtifactGate-EDA addresses this problem as a software engineering
method and tool for preparing, validating, replaying, and auditing EDA
experiment artifacts. The tool indexes artifacts through adapters, records file
hashes and provenance, assigns bounded evidence levels, checks claims against a
policy, builds replay manifests, detects corrupted packages, and emits
reviewer-facing reports. We evaluate ArtifactGate-EDA with multi-adapter
ingestion, replay reproducibility, ClaimBench-EDA claim checking, corrupted
artifact detection, evidence-level classification, scalability, baseline
comparison, ablation, and optional-backend audit experiments. The evaluation
covers 1300 policy-seeded ClaimBench-EDA rows, 1000 evidence-level
gold-standard rows, 900 corrupted-artifact instances plus 30 clean specificity
cases, 10 public external EDA cases with 8 open-source ArtifactGate
replay-package/validation cases, seven scalability levels up to 100k synthetic
manifest rows, seven baseline/target emulators, and ten ablation variants. The
package also includes a 16-row generated reviewer-walkthrough dry run for
author-side protocol preparation; this is not used as human or expert-study
evidence. The results show that the method can make
software-only EDA artifact packages claim-safe, replay-checkable, and
reviewer-auditable without requiring proprietary tools or device-side evidence
as core reproduction dependencies.

ArtifactGate-EDA is a software engineering method and tool for making
software-only EDA experiment artifacts claim-safe, replay-checkable, and
reviewer-auditable.

## 1. Introduction

Reproducibility in EDA research is not only a question of whether files are
available. Reviewers also need to know what those files prove, what they do not
prove, and whether manuscript claims stay within the evidence layer represented
by the shared artifacts. A SPICE log, an HDL simulation transcript, and a Yosys
synthesis report support different types of statements. Without an explicit
artifact-to-claim contract, a package can appear reproducible while still
mixing software simulation evidence with stronger claims that require evidence
not present in the package.

ArtifactGate-EDA treats this as a software engineering problem. The goal is to
turn artifact preparation into a structured workflow with adapters, schemas,
hashes, replay manifests, evidence levels, claim policies, and report
generation. The method is intentionally limited to software-only EDA artifacts.
It does not attempt to validate circuit correctness in general, prove complete
verification, or replace tool-specific design flows. Instead, it provides a
claim-safety layer around the artifacts that are actually included in a
submission package.

The contributions of this work are:

1. A claim-safe artifact engineering model for software-only EDA experiments.
2. An adapter-based implementation covering SPICE/ngspice, HDL simulation,
   Yosys synthesis, Verilator examples, metadata-only optional adapters, and a
   schema-only vendor boundary adapter.
3. A policy-driven claim checker that reports unsupported or evidence-escalated
   statements and proposes bounded rewrite suggestions.
4. An empirical evaluation suite covering ingestion, replay, claim injection,
   corrupted packages, evidence-level classification, external case
   generalization, scalability, baselines, ablation, and optional-backend
   behavior, plus a generated reviewer-walkthrough dry run for author-side
   protocol preparation.
5. A reproducible report package generated by `make ist-all`.

## 3. Method

ArtifactGate-EDA represents each included file as an artifact record. A record
contains a relative path, artifact type, SHA-256 hash, tool label, tool version
field, adapter name, evidence level, claim binding, unsupported boundary, and
creation command. The method has five stages.

First, adapter ingestion maps files from a case directory into a normalized
artifact index. The adapter determines which file extensions are relevant and
assigns initial evidence levels. Second, validation checks file existence,
hashes, tool-version fields, portable paths, evidence-level fields, and claim
references. Third, replay packaging creates a replay manifest and a validation
report for software-only examples. Fourth, claim checking applies forbidden
claim and evidence-level policies to manuscript or package claims. Fifth,
report generation emits tables and reviewer-readable Markdown summaries.

The evidence-level model deliberately separates metadata, source presence,
reference/interface records, software simulation, and synthesis evidence. The
highest positive evidence layer in the current core package is
`L4_SYNTHESIS`. Stronger device-side evidence layers are represented only as
unsupported boundaries unless future artifacts provide direct evidence and a
separate review gate accepts them.

## 4. Tool Implementation

The tool is implemented as a Python package with a command-line interface named
`artifactgate`. The IST version exposes these commands:

- `artifactgate ingest`
- `artifactgate validate`
- `artifactgate replay`
- `artifactgate claim-check`
- `artifactgate report`
- `artifactgate package`
- `artifactgate compare`
- `artifactgate benchmark`
- `artifactgate ablate`

The implementation keeps deterministic report generation inside the package so
that evaluation outputs can be regenerated by Makefile targets. The
`benchmark` command runs repository-quality, corrupted-artifact,
evidence-classification, external-case, scalability, baseline, backend-audit,
and report summary suites. The `ablate` command generates the component
ablation table.

The package includes core examples for ngspice, Icarus, Yosys, and Verilator;
metadata-only examples for PLECS, LTspice, and Logisim; and a schema-only vendor
boundary example. Optional local executables are audited, but their presence is
not required for core reproduction.

## 5. Evaluation Design

The evaluation is organized around nine empirical research questions plus one
generated dry-run preparation item.

RQ1 evaluates multi-adapter ingestion across core, optional, and boundary
adapters. RQ2 evaluates replay reproducibility for software-only examples,
including 10 repeated replay runs for each core replay case and artifact-signature
drift checks. Local Docker execution is recorded separately because the current
machine does not provide a Docker binary. RQ3
uses a 1300-row policy-seeded ClaimBench-EDA dataset with safe software-only
claims, unsupported hardware/Vivado/DFX/bitstream overclaims, ambiguous
external-evidence claims, paraphrase hard cases, and safe limitation statements.
The ClaimBench-EDA gold standard is deterministic and policy-seeded rather than
independently human-rated. RQ4 evaluates a
30-class corrupted-artifact suite with 30 instances per defect class and 30
clean specificity cases. RQ5 uses
a 1000-row evidence-level gold standard with policy-development, holdout, and
cross-tool holdout groups. E6/G9 evaluates 10 public external EDA cases,
including 8 open-source ArtifactGate replay-package/validation cases and 2
metadata-only Logisim cases, without requiring hardware or proprietary tools.
The scalability extension evaluates 1k, 3k, 5k, 10k, 30k, 50k, and 100k
synthetic manifest rows, with at least five repeats up to 50k and three
repeats at 100k. RQ7 compares ArtifactGate-EDA with manual and lightweight
baselines. RQ8 evaluates component ablations. RQ9 audits optional local
backends and fallback behavior.

The primary acceptance condition is that all RQ reports are generated and that
critical claim-boundary failures remain zero. The full command is:

```bash
make ist-all
```

## 6. Results

The current evaluation reports are generated under `reports/`.

RQ1 generated `reports/rq1_multi_adapter_ingestion.md` and CSV metrics for
ngspice, Icarus, Yosys, Verilator, PLECS, LTspice, Logisim, and the boundary
adapter. RQ2 generated replay reports for ngspice minimal, CircuitFaultBench
sample, Icarus, Verilator, and Yosys cases; the current aggregate is
`reports/rq2_replay_reproducibility.md` with case-level CSV data in
`reports/rq2_replay_summary.csv`. The repeatability extension completed 10
repeats for each of the five replay cases with no artifact-signature drift in
`reports/rq2_replay_repeats.md` and
`reports/rq2_replay_repeat_summary.csv`; Docker/CI availability is recorded in
`reports/rq2_docker_ci_status.csv` without claiming a local Docker pass. RQ3 evaluated 1300 ClaimBench-EDA rows and
reported unsupported recall 1.00, macro-F1 1.00, safe-limitation false-positive
rate 0, safe-rewrite coverage 1.00, and zero critical false negatives in
`reports/rq3_claimbench_summary.md`,
`reports/rq3_confusion_by_claim_type.csv`, and
`reports/rq3_claimbench_results.csv`. RQ4 evaluated a deterministic
software-only corruption benchmark with 900 corrupted-artifact instances across
30 defect classes, plus 30 clean specificity cases, and reported zero critical false passes in
`reports/rq4_corrupted_artifact_detection.md` and
`reports/rq4_error_classification_matrix.csv`. The extended taxonomy, instance
results, clean specificity set, and severity-weighted score are reported in
`reports/rq4_defect_taxonomy.md`,
`reports/rq4_corruption_extended_results.csv`,
`reports/rq4_clean_specificity_cases.csv`, and
`reports/rq4_severity_weighted_detection.md`. RQ5 classified 1000
gold-standard rows with holdout and cross-tool holdout labels, accuracy 1.00,
macro-F1 1.00, unsupported-level recall 1.00, and zero critical escalation errors in
`reports/rq5_evidence_level_classification.md` and
`reports/rq5_evidence_level_confusion_matrix.csv`; split metrics are recorded
in `reports/rq5_evidence_level_holdout_summary.csv`. RQ6 completed all four
external-case thresholds: 10 public cases, 8 open-source ArtifactGate
replay-package/validation cases, adapter success rate 1.00, observed manual
adapter fix rate 0, and observed hardware dependency count 0 in
`reports/rq6_external_case_generalization.md`,
`reports/rq6_external_case_generalization.csv`, and
`reports/rq6_external_case_generalization_summary.csv`. The scalability
extension wrote and row-parsed synthetic ArtifactGate manifests up to 100k
rows, with 53 measured manifest-processing runtime observations across seven
scales and linear-fit R2 above 0.99 in
`reports/rq7_scalability_summary_extended.md`,
`reports/rq7_scalability_extended_runtime.csv`,
`reports/rq7_scalability_extended_memory.csv`, and
`reports/rq7_scalability_model_fit.csv`; the fit plot is
`paper/figures/scalability_fit.png`. The baseline extension ran seven
deterministic baseline/target emulators on one shared software-only fixture.
Each emulator wrote method-specific outputs under `outputs/rq7_baseline/`, and
the evaluator read those generated outputs to produce 56 task observations in
`reports/rq7_baseline_task_results.csv` and summary metrics in
`reports/rq7_baseline_task_execution.md` and
`reports/rq7_baseline_comparison.md`. RQ8 generated ten ablation variants with
observation-level rows, bootstrap confidence intervals, and effect sizes in
`reports/rq8_ablation.md`, `reports/rq8_ablation_results.csv`,
`reports/rq8_ablation_effect_sizes.csv`, and
`reports/rq8_ablation_observations.csv`. RQ9 generated an optional-backend status
table with fallback behavior in `reports/rq9_local_backend_audit.md` and
`reports/rq9_optional_backend_status.csv`. RQ10 generated a reviewer-walkthrough
dry run with manual-package and ArtifactGate-package conditions, 16 task
observations, and command-log timestamps in
`reports/rq10_reviewer_walkthrough.md`,
`reports/rq10_reviewer_walkthrough_observations.csv`, and
`reports/rq10_reviewer_walkthrough_command_log.csv`. This is preparatory
material for an author-side expert walkthrough, not completed expert
walkthrough evidence, a multi-participant human-subject study, or measured
human timing result.

The overall result is summarized in
`reports/ist_empirical_evaluation_summary.md`, where all required RQ reports
are present.

## 7. Discussion

The evaluation supports the central software engineering claim: ArtifactGate-EDA
adds value beyond file bundling by linking artifacts to hashes, evidence
levels, replay manifests, claim checks, defect classifications, and
reviewer-facing reports. Within the deterministic baseline-emulator harness,
plain archives, checksum lists, shell scripts, and simple manifests do not
provide the same combination of output-level claim-boundary checking and
evidence-level reporting. The
ablation study shows that the claim policy, evidence-level model, hash checks,
metadata extraction, replay manifests, and safe rewrite suggestions each
contribute to the final package quality.

The method is most useful when a project already has software artifacts but
needs to prevent overclaiming during paper preparation or artifact review. It
does not remove the need for domain review. It makes the evidence boundary more
explicit so that authors and reviewers can identify when stronger evidence
would be needed.

## 8. Threats to Validity

The claim dataset is generated from controlled templates, so it may not cover
all linguistic forms that appear in real submissions. The corrupted-artifact
suite is deterministic and focuses on package-level defects rather than every
possible tool-specific failure mode. The scalability benchmark uses synthetic
artifact rows to evaluate package-processing behavior, not physical design
complexity. The scalability memory values are estimates derived from manifest
size and row count; they are not measured process RSS. The baseline benchmark
uses a deterministic task harness and operator-step time estimates, not a
human-subject timing study. The external-case generalization benchmark uses public source
files and ArtifactGate replay-packaging/validation or metadata-only ingestion
checks; it does not claim independent EDA tool execution, circuit correctness,
hardware validation, or vendor timing closure. The baseline and ablation
results are designed to compare software engineering capabilities, not to
benchmark EDA algorithms. The generated reviewer-walkthrough dry run is not
evidence that the optional G13 expert walkthrough has been completed;
participant or expert-walkthrough claims require separate author-confirmed
human data.

The evaluation mitigates these threats by publishing the datasets, scripts,
Makefile targets, and generated reports. Future work should add claims from
independent manuscripts, more artifact formats, and externally reviewed
corruption cases.

## 9. Limitations

ArtifactGate-EDA checks whether artifact packages support bounded claims. It
does not prove that a circuit, controller, HDL module, or EDA flow is correct.
It does not make proprietary tools a core reproduction requirement. It does not
convert software simulation or synthesis artifacts into stronger device-side
evidence. The current evidence ceiling is `L4_SYNTHESIS`.

Optional backend audit results depend on the local environment and are reported
as optional. Missing optional executables are handled by metadata-only or
schema-only fallback behavior.
